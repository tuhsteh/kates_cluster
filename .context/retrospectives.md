### log-ramdisk: Mount /var/log as tmpfs ramdisk
- **Avoid**: Each `bash` tool call runs in a fresh shell — `git checkout -b feature/log-ramdisk` did not persist to a later commit call, so the commit landed on `main`. Fix: always include `git checkout <branch>` in the same bash call as the git operation (e.g., `git checkout feature/log-ramdisk && git commit ...`), or verify branch with `git branch` immediately before committing.
- **Repeat**: @reviewer caught a silent correctness risk (journald switching to persistent mode on the tmpfs, filling the ramdisk unexpectedly) that wasn't obvious from the implementation alone. Worth the review pass even on small new roles.
- **Updated**: nothing to promote — no committed `.context/` in this project (globally gitignored). If `initialize-repo` is run in future, promote the journald/tmpfs pattern to `domains/raspberry-pi.md`.

### k3s-ssd-data-dir: k3s data-dir and kubelet root-dir on SSD with systemd mount ordering
- **Avoid**: When relocating a service's data directory, grep the entire role for **all** hardcoded references to the old default path before committing. The coder added new tasks and templates but did not audit existing tasks — token path (`/var/lib/rancher/k3s/server/token`) was hardcoded in 3 places and would have caused `wait_for` to time out on the first full run. Fix: always `grep -r <old-default-path> roles/<role>/` after any data-dir config change.
- **Repeat**: Extract a single role default variable (`k3s_data_dir`) and use it everywhere in tasks, templates, and handlers. This keeps the path DRY and makes the class of drift-bug structurally impossible. The reviewer suggested it; the fix was straightforward because all references were in one role.
- **Updated**: Appended k3s path inventory, `/var/lib/kubelet` exclusion explanation, systemd drop-in pattern, and SSD mount guard to `.context/domains/raspberry-pi-hardware.md`.

### ssd-fstab-mount: UUID-based m.2 SSD discovery and persistent mount
- **Avoid**: `ansible.builtin.command` runs unconditionally in `--check` mode. When it follows a `community.general.filesystem` task (which correctly skips in check mode), it operates on an unformatted device, returns empty output, and triggers any UUID-empty fail guard — producing a misleading fatal on dry-runs of unprovisioned nodes. Fix: gate the command and its dependents with `when: not ansible_check_mode`; add a placeholder set_fact for the check-mode path. Promoted to `.context/domains/raspberry-pi-hardware.md`.
- **Repeat**: Researcher-first for hardware-specific discovery paid off — surfaced the CM4 `rotational: "0"` gotcha (both SD cards and SSDs report non-rotational) before any code was written, avoiding a hard-to-debug silent mis-identification. Worth dispatching @researcher whenever Ansible meets unfamiliar hardware.
- **Updated**: Created `.context/domains/raspberry-pi-hardware.md` with CM4 device naming, NVMe-first discovery pattern, check-mode safety pattern, and `ansible.posix.mount` fstab gotchas.
